## 一、背景与目标

本群汇集了多个飞书 AI 助手机器人，为避免功能重叠、消息冲突和重复响应，制定统一桥接规则，建立有序协作机制。

**目标：**

- 明确各 bot 能力边界与职责分工
- 建立消息路由规则，确保用户请求被最优 bot 响应
- 形成可迭代的规则版本管理机制
- 保障规则讨论独立于群消息，不干扰日常使用

## 二、方案架构（方案C - 双轨制）

基于 ZongliProMaxUltra 提出的方案C，采用双轨制运行：

### 轨道一：执行轨（当前群）

- 用户在群内直接 @ 某个 bot 或通用提问
- 按 routing 规则自动路由到最优 bot
- 规则执行对用户透明，无需感知底层逻辑

### 轨道二：迭代轨（独立群/任务）

- 规则的讨论、修订、版本管理在独立通道进行
- 通过飞书任务跟踪迭代进度
- 版本变更需经过评审后合入主线

## 四、路由规则

### 4.1 显式路由

用户 @ 某个 bot 时，由该 bot 直接响应。

✅ **帆½ 触发说明**：帆½ 仅响应**显式 @帆½**（代码级结构化 mention 强制校验）；消息文本中仅出现「帆½」字样但未真正 @，不会触发。未 @ 帆½ 时帆½ 保持静默，不进入隐式路由。

### 4.2 隐式路由（通用提问）

当用户未指定 bot 时，按以下优先级匹配：

1. **飞书工作流操作**（文档/云盘/日历/任务/消息）→ fufu 001
2. **绩效/报告生成** → ZZZJIN
3. **方案设计/架构分析** → ZongliProMaxUltra
4. **通用问答** → 最先响应的 bot，其余 bot 保持静默

### 4.3 冲突避免

- 同一请求只允许一个 bot 响应（显式 @ 除外）
- 首个响应的 bot 「锁定」该请求，其他 bot 在 30 秒内不重复响应
- 若首个响应 bot 超时未完成，其他 bot 可接管

## 五、迭代机制

### 5.1 版本管理

- 主线版本存放于飞书云盘 `/桥接规则/` 目录
- 版本号格式：v{major}.{minor}，如 v0.1
- 每次修订递增 minor，重大架构变更递增 major

### 5.2 迭代流程

1. 提出修改建议（在独立迭代群或飞书任务）
2. 讨论评审（维护人参与）
3. 合入主线版本，更新版本号
4. 群内通知版本变更摘要

### 5.3 任务跟踪

通过飞书任务跟踪以下事项：

- 各 bot 能力 Card 收集
- 路由规则测试
- 轮值机制探索
- 版本迭代评审

## 六、独立性保障

- 规则**执行**在当前群按规则自动运行
- fufu 001 作为执行者，按规则文档运作，迭代事项转发到独立通道

# 九、校验结果与缺陷修复

2026-07-14，42号机器人对桥接规则 v0.1 进行了校验测试。校验结论：**桥接规则存在关键缺陷，需修复后方可进入 v0.2 迭代。**

## 9.1 达标项

| 校验点 | 说明 |
|-|-|
| 意图分类正确 | 正确识别为查询，未误判为收录意图 |
| 周报边界清晰 | 回复中说明「不会计入周报」 |
| 未编造结果 | 如实报告未找到相关信息 |
| 主动澄清 | 向用户提出澄清问题 |

## 9.2 未达标项

| 缺陷 | 根因 | 影响 |
|-|-|-|
| 任务解析误收录 | Parse-ReplyTasks 匹配条件 "\$Reply -match 已收录" 过宽，回复中「没有已收录的任务记录」触发了误匹配 | 两条澄清问题被错误写入 tasks.json |
| 缺少回滚机制 | 误收录后无自动清理或回滚流程 | 污染后续上下文 |
| 否定语境未排除 | 「已收录」三字匹配条件覆盖了否定语境 | 非收录回复被当作收录处理 |

## 9.3 修复方案

**修复1：收紧匹配条件。**将任务收录的匹配关键词从「已收录」收紧为「已收录到本周」或「已收录到本周周报素材」，避免否定语境中的「已收录」触发误匹配。

**修复2：增加上下文校验。**匹配到收录关键词后，额外检查上下文是否为肯定语句（排除「没有」「未」「不」等否定词前缀），双重确认后才执行收录。

**修复3：增加回滚机制。**每次收录操作记录时间戳和内容快照，发现误收录时可一键回滚。详见第十章。

# 十、回滚与清理机制

## 10.1 收录操作日志

每次任务收录操作记录以下信息，存储于操作日志中：

- 操作时间（毫秒时间戳）
- 触发来源（用户消息 ID + 消息内容摘要）
- 收录内容快照（写入 tasks.json 的原始数据副本）
- 操作结果（成功 / 失败）

## 10.2 回滚流程

1. 检测到误收录（通过自动校验或人工发现）
2. 从操作日志中获取内容快照
3. 从 tasks.json 中删除对应记录
4. 在群内发送回滚通知，说明误收录原因和已清理内容
5. 更新校验规则，避免同类问题再次发生

## 10.3 防护措施

- **收录前预检：**匹配关键词 + 上下文否定词检测，双重确认
- **收录后校验：**写入后 30 秒内自动校验内容合理性（是否为澄清问题、是否为否定语句等）
- **人工兜底：**任何成员可在群内 @fufu 001 指定回滚，机器人收到指令后立即执行清理

# 十一、帆½ 实践方案（Codex → WorkBuddy 迁移）【🆕新增】

> 本章基于帆½ 实际落地方案，补充 v0.1 中「帆½ 待登记」所缺失的实现细节。原 Codex 引擎因依赖梯子导致反馈延迟，已迁移至 WorkBuddy 作为回复大脑。

## 11.1 架构与运行形态

| 组件 | 形态 | 职责 | 时效 |
| --- | --- | --- | --- |
| Gateway `bridge.py` | 本地 launchd `com.yousandi.hmi-ai-workers-bridge`，`--mode hybrid --interval 10` | 监听飞书（`lark-cli event consume` 推送 + 轮询兜底）、去重、加 `OK`、登记任务、发预览、处理确认发布 | 常驻 |
| 大脑（文本兜底）`wb_bridge_fastpoll.py` | 本地 launchd `com.yousandi.wb-bridge-fastpoll`，每 120s | 调用 **DeepSeek API**（`deepseek-chat`）生成文本回复草稿 → 发私聊预览 | ≤ 120s |
| 大脑（多模态/后备）WorkBuddy 自动化 | HOURLY | 处理多模态任务，以及 DeepSeek 失败时的兜底 | ≤ 1h |

- Gateway 直连 `open.feishu.cn`，**不依赖梯子/VPN**，不调用任何大模型。
- 文本任务：优先由 DeepSeek fastpoll 即时生成（低成本）。
- 多模态任务 / DeepSeek 失败：交给 WorkBuddy 自动化（每小时）。
- ✅ **多模态窗口已闭合**（方案①，2026-07-14 落地）：原任务 10 分钟过期（`eta_at = created + 600s`）会导致 WorkBuddy 每小时跑必然错过多模态窗口。修法仅改网关 launchd 环境变量（**不动 `bridge.py` 一行代码**，重启网关一次）：`HMI_BRIDGE_ESTIMATED_REPLY_SECONDS` 600→5400（90 分钟）、`HMI_BRIDGE_MAX_BACKLOG_SECONDS` 3600→5400。多模态任务现可存活至下一次 WorkBuddy 运行，缺口闭合。代价：多模态最坏延迟 ≈ 1h（仅多模态；文本仍 ≤120s）。
